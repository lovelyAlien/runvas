# 협업 가이드

이 프로젝트는 2명이 백엔드와 모바일을 나눠 작업하는 것을 전제로 합니다.

## 추천 역할 분담

### 담당자 A: Backend

- 코스 생성 API
- 코스 목록/상세 조회 API
- bounds 기반 검색
- 거리 및 예상 소요 시간 계산
- GPX 다운로드 응답
- 인증 기초 구조
- 공개 코스 복사 API
- 게시글/댓글 CRUD API
- 코스/게시글 좋아요 API
- 코스/게시글/댓글 신고 API
- 작성자 권한 검증
- 커뮤니티 목록 페이지네이션과 인기순 정렬

### 담당자 B: Mobile

- 지도 화면
- 경로 그리기 인터랙션
- 코스 작성 폼
- 코스 탐색 목록/상세 화면
- 로그인/프로필 화면
- 커뮤니티 게시글 목록/상세 화면
- 댓글 작성 화면
- 좋아요/신고 인터랙션
- API 연동
- GPX 다운로드 UX

## 첫 구현 순서

1. `runvas-docs`에서 Course 모델과 API 계약 확정
2. `runvas-docs`에서 커뮤니티 모델과 API 계약 확정
3. `runvas-backend`에서 mock 또는 in-memory 저장소로 코스 API 먼저 구현
4. `runvas-backend`에서 인증, 게시글, 댓글, 좋아요, 신고 API를 mock 또는 in-memory로 구현
5. `runvas-mobile`에서 mock API 또는 개발 서버 API로 코스 화면을 연동
6. `runvas-mobile`에서 커뮤니티 화면을 연동
7. bounds 조회, 인기순 정렬, GPX 다운로드를 실제 구현
8. 인증과 영속화 세부 구현

## 계약 변경 절차

API나 데이터 모델 변경이 필요하면 다음 순서로 진행합니다.

1. `runvas-docs` 문서를 먼저 수정합니다.
2. 백엔드와 모바일 담당자가 변경 내용을 확인합니다.
3. 백엔드 구현을 수정합니다.
4. 모바일 구현을 수정합니다.
5. 양쪽에서 같은 예시 요청/응답으로 동작을 확인합니다.

## 완료 기준

공통 계약 변경은 다음 조건을 만족해야 완료로 봅니다.

- 문서에 요청/응답 예시가 있다.
- 필드 타입과 필수 여부가 명확하다.
- 좌표 단위와 순서가 명확하다.
- 백엔드와 모바일 중 어느 쪽이 계산하거나 검증하는지 명확하다.
- 작성자 권한, 공개 범위, 중복 좋아요/신고 처리 방식이 명확하다.
